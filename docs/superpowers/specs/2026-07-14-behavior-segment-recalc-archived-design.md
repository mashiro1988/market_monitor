# 段重算脚本 存档设计（出事再建）

> 2026-07-12 决策"历史段修正走显式重算脚本"的定稿设计。2026-07-13~14 与用户多轮拍板后**决定不现在实现**：
> 生产 39 天 BTC 零补写零缺口；游标同步重构（同日落地）后，事后写入历史的唯一来源只剩"停机 >6h 后重启"。
> 本文档是出事那天的施工图，按此实现半天可落地。

## 解决什么

R2（settle 写保护）后历史段只读：`behavior_cycle` 只写"结束时间在最近 6h 内"的段。若停机 >6h 后重启，
启动首轮扫描会把停机窗口的 bar 写进历史（游标同步窗口自愈），但该区域内的段**永远不会被记录**
（在线周期对 6h 外无笔）——需要本脚本一次性补检/修正。

## 设计四要点（全部已拍板）

1. **识别（无状态 CDC）**：扫最近 7 天入库的 BTC/USDT bar，`created_at − timestamp > 6h` 即"脚本管辖的事后 bar"
   （更晚近的在线周期自己修；30min 只是"正常写入延迟"的概念线，脚本只看 6h 分工线 = WRITE_HORIZON_HOURS）。
   无水位表：对账幂等使重复处理=无操作，漏跑下轮覆盖。
2. **作用域（包络 + 安静间隙）**：取事后 bar 的 [min, max] 全集包络，向两侧扩到 **≥40min 无基档触发**的
   安静间隙为止（链式合并上限 = 15min 回看 + 10min 容差 + 5min 合并 = 30min，40min 必断链，永不拦腰）。
   不逐洞聚簇——对账幂等才是写作用域的真正护栏（detect_segments 是纯函数，数据没变的区域重检必然一致）。
   若包络右缘仍在 now−6h 写权区内 → 本次不动，等滑出再跑。
3. **对账（幂等，用户拍板的简化版）**：作用域内重检测后——边界与旧行分毫不差**且**段内无事后 bar → 原行
   一字不动（human_class 保留）；其余（形状变/被吸收/新出现/段内有新 bar）→ 删旧插新，`human_class` 一律
   置空 → 段自动回标注工作台待标队列（用户日常审核时重标）。不做重叠继承。新段直接走完整 settle
   （S 分数/新闻命中/classification 按完整数据现算）。
4. **收尾**：受影响 `utc_date` 重跑 `aggregate_day` 追加 PIT 日汇总（表设计天然支持）；WeCom 简报
   （"重算 X~Y：M 段→N 段，K 条人工标注回待标"）。

## 触发方式

手动：`python scripts/recalc_behavior_segments.py [--since-days 7]`（建议放 scripts/）。
何时跑：停机 >6h 重启后、或手动深回补历史 bar 后。可选升级：cron 每小时（无状态设计零改动即可挂）。
